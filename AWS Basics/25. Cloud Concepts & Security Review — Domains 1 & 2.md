# Cloud Concepts & Security Review — Domains 1 & 2

## Overview

This is a study guide for AWS Certified Cloud Practitioner Domains 1 and 2, covering foundational cloud concepts and security. These two domains comprise 51% of the exam (26% + 25%). Understanding them well is non-negotiable for passing.

This guide connects the 19 technical topics covered earlier in the series to their exam relevance and bundles them with common exam question patterns.

---

## Domain 1: Cloud Concepts and Technology (26% of exam, ~39 questions)

### Cloud Deployment Models

| Model | Infrastructure location | Responsibility | AWS service | Use case |
|---|---|---|---|---|
| Public Cloud | AWS datacentres | Customer configures, AWS provides infrastructure | All AWS services | Most workloads; scalability; elasticity |
| Private Cloud | On-premises | Customer owns everything | AWS Outposts | Compliance; data residency; control |
| Hybrid Cloud | On-premises + AWS | Split responsibility | AWS Outposts, Direct Connect, Storage Gateway | Legacy + cloud migration |

**Exam focus:** Understand when each is appropriate. Exam often asks: 'A company wants flexibility and cost savings but cannot move data off-premises.' Answer: hybrid (AWS Outposts, Direct Connect, Storage Gateway).

### Cloud Service Models (IaaS vs PaaS vs SaaS)

This is foundational and critical. Many exam questions hinge on this.

| Model | Customer manages | AWS manages | Examples | Responsibility for security |
|---|---|---|---|---|
| **IaaS** (Infrastructure as a Service) | OS, runtime, app, data, access | Hardware, hypervisor, network | EC2, EBS, VPC | Customer owns most of the stack |
| **PaaS** (Platform as a Service) | App code, data, access | OS, runtime, middleware, database engine | RDS, Elastic Beanstalk, Lambda | Shared: customer owns app and access; AWS owns platform |
| **SaaS** (Software as a Service) | Data, access | Everything else | S3 (accessed via web console), DynamoDB (via API) | AWS owns all infrastructure; customer owns data |

**Exam tip:** Questions ask which service is which model. EC2 = IaaS. RDS = PaaS. S3 = SaaS (you interact via an API, not manage servers). Lambda = PaaS (you write code, AWS runs it).

### Regions and Availability Zones

| Concept | Definition | Relevant topics |
|---|---|---|
| **Region** | Geographical area containing multiple AZs; completely isolated from other regions | Route 53 failover, CloudFront, S3 cross-region replication |
| **Availability Zone** | Physically separate datacentre within a region with independent power, cooling, networking | Multi-AZ RDS, ALB spanning AZs, ASG across AZs |
| **Edge Location** | Cache node for CloudFront; different from AZs | CloudFront for global caching |

**Exam mistake:** Confusing Multi-AZ (within one region, protects against datacentre failure) with global deployment (multiple regions, protects against region failure). Multi-AZ does NOT provide lower latency globally.

**Question pattern:** 'To reduce latency for users in India and Singapore, which service should we use?' Answer: CloudFront (edge locations), not Multi-AZ.

### AWS Cost Model

| On-premises | AWS |
|---|---|
| Large upfront capex (servers, cooling, staff) | Pay-as-you-go (per vCPU-hour, per GB stored) |
| Fixed cost regardless of usage | Variable cost; scale up and down |
| Difficult to right-size (over-provision for peak) | Easy to right-size; only pay for what you use |
| Cannot easily dispose of unused capacity | Stop instances, delete storage; cost drops to zero |

**Exam focus:** Cost model advantages of cloud: elasticity, no capex, pay-for-what-you-use, ability to reduce cost by turning things off.

### Scalability and Elasticity

**Scalability:** the ability to grow capacity to meet demand. Horizontal (add more instances) or vertical (bigger instance).

**Elasticity:** automatically scaling up and down in response to demand changes. Requires automation (Auto Scaling Group).

Elasticity is a subset of scalability. All elastic systems are scalable; not all scalable systems are elastic.

**Exam question pattern:** 'Which AWS feature automatically adjusts capacity based on traffic?' Answer: Auto Scaling (elasticity), not just having the ability to add instances (scalability).

## Domain 2: Security, Compliance, and Governance (25% of exam, ~38 questions)

### AWS Shared Responsibility Model (Critical)

This is the foundation of AWS security. Miss this and you lose ~5-10 exam questions.

**AWS is responsible for (Security OF the Cloud):**
- Physical datacentres, hardware, hypervisor
- Network infrastructure, DDoS mitigation
- Managed service runtimes (RDS patching, Lambda runtime updates)
- Global infrastructure and availability

**Customer is responsible for (Security IN the Cloud):**
- IAM: users, roles, policies, access control
- OS-level security: EC2 instance patching, firewall rules (security groups, NACLs)
- Application security: code vulnerabilities, dependencies, input validation
- Data: encryption decisions, retention, deletion, access logging

**The boundary shifts by service type:**
- EC2 (IaaS): customer owns most; AWS owns hardware
- RDS (PaaS): AWS patches the DB engine; customer owns access control and data
- S3 (SaaS): AWS owns everything below the API; customer owns bucket policy and data

**Exam pattern:** 'Who is responsible for patching the database OS on an RDS instance?' Answer: AWS (it is a managed service). If it were EC2 with a self-managed database: customer.

### Identity and Access Management (IAM)

**Core concepts:**
- **Users**: represent individual humans or applications; long-term access keys
- **Roles**: temporary credentials for services or cross-account access; preferred for applications
- **Policies**: JSON documents that define permissions (what action, on what resource)
- **MFA**: multi-factor authentication; required on root account
- **Least Privilege**: grant only the minimum permissions needed

**Exam focus:**
- Root user should have MFA enabled; root should not be used for daily work
- Services (EC2, Lambda) should assume roles, not use access keys
- Roles can be assumed by other AWS accounts (cross-account access)

**Question pattern:** 'A Lambda function needs to read from S3. What is the best practice?' Answer: Create an IAM role with s3:GetObject permission and attach it to the Lambda execution role.

### Encryption

**At Rest (stored data):**
- SSE-S3: S3 manages keys; simplest, no additional cost
- SSE-KMS: customer manages keys via AWS KMS; more control
- Client-side encryption: customer encrypts before sending to AWS

**In Transit (data moving over network):**
- TLS/SSL (HTTPS): encrypts data between client and server
- All AWS services support HTTPS; it should be enforced

**Exam pattern:** 'An organisation wants to ensure data is encrypted at rest in S3. What is the minimum configuration needed?' Answer: Enable S3 default encryption (SSE-S3 or SSE-KMS).

### AWS Compliance and Certifications

**Key compliance frameworks:**
- SOC 2 (System and Organization Controls): audited controls for security, availability, confidentiality
- ISO 27001: information security management
- PCI DSS: payment card data security
- HIPAA: healthcare data privacy (requires Business Associate Agreement)
- FedRAMP: US federal government cloud authorisation

**AWS Artifact:** AWS's self-service portal where you download compliance reports (SOC 2, ISO 27001, PCI DSS). Proves AWS has done its job on the infrastructure layer.

**Exam focus:** Know that AWS publishes compliance reports via Artifact. Know the difference between AWS compliance (infrastructure) and customer compliance (their application and data).

### AWS Organizations

**Organizational structure:**
- Root: the master account
- Organisation Units (OUs): logical groupings of accounts
- Policies: SCPs (Service Control Policies) apply at the OU level; restrict maximum permissions for all accounts beneath

**Use case:** Prevent production accounts from deleting RDS instances, or restrict s3:DeleteBucket across all non-prod accounts.

**Exam focus:** Understand that AWS Organizations provides consolidated billing and centralized policy enforcement across multiple accounts.

## Key Exam Patterns and Traps

### Pattern 1: Service Model Questions

**Question:** 'Which of the following is a Platform as a Service (PaaS) offering?'

Options: EC2, RDS, S3, CloudFront

**Answer:** RDS. EC2 = IaaS, S3 and CloudFront = SaaS.

### Pattern 2: Responsibility Boundary Questions

**Question:** 'A company runs a web application on EC2 instances behind an ALB. After a security audit, they discover an EC2 instance is missing critical OS patches. Who is responsible for patching the OS?'

Options: AWS, Customer, both

**Answer:** Customer. EC2 is IaaS; patching the OS is the customer's responsibility. (If it were RDS, the answer would be AWS.)

### Pattern 3: Scalability vs Elasticity

**Question:** 'A startup wants to reduce infrastructure costs during off-peak hours. Which AWS feature supports this?'

Options: ALB, CloudFront, Auto Scaling, Multi-AZ

**Answer:** Auto Scaling (elasticity). ALB distributes traffic, CloudFront caches globally, Multi-AZ provides failover. Only Auto Scaling scales based on demand.

### Pattern 4: Regional vs Global

**Question:** 'A company has users across North America and Europe. To minimize latency, which service should they use?'

Options: Multi-AZ RDS, CloudFront, S3 standard, Elastic Load Balancer

**Answer:** CloudFront (global edge locations). Multi-AZ is regional. Standard ELB is regional. S3 is regional (though can be accessed globally).

## Practice Quiz Strategy

### Before the Quiz

1. Review the list of exam topics for Domains 1 & 2
2. Spend 5 minutes on each key concept above
3. Note which concepts feel weak

### During the Quiz

1. Time yourself: 30 questions in 45 minutes (1.5 minutes per question)
2. Do not second-guess; first instinct is usually correct
3. Eliminate obviously wrong answers first
4. Watch for scenario-based wording: 'A company wants to...' — context matters

### After the Quiz

1. Score: aim for 80%+ (24+ out of 30 correct)
2. Review every wrong answer:
   - What concept did you miss?
   - Does it relate to Shared Responsibility, service models, or regional vs global?
   - Write down the pattern
3. Do not retake the same quiz immediately; study the gaps first

## Common Exam Mistakes (What You'll Get Wrong)

### Mistake 1: Confusing Multi-AZ with Global

Multi-AZ = two datacentres in the same region (protects against one datacentre failure, not region failure).

Global = CloudFront, Route 53, S3 cross-region replication (protects against region failure, adds latency benefits).

**Wrong answer on exam:** 'To serve users globally with low latency, enable Multi-AZ RDS.' Correct: 'Use CloudFront or S3 cross-region replication.'

### Mistake 2: Forgetting Shared Responsibility Context

Answering 'AWS is responsible' for something that requires customer configuration (e.g. 'Who is responsible for security group configuration?').

Always think: is this about the infrastructure layer (AWS) or the configuration/application layer (customer)?

### Mistake 3: Assuming SaaS Means No Customer Security Responsibility

S3 is SaaS (AWS manages servers, networking, durability). But the customer is still responsible for:
- Bucket policies (who can access)
- Encryption (whether to use SSE-KMS)
- Data retention and deletion
- Logging and monitoring

AWS's SaaS coverage is infrastructure-only; customer security responsibility remains.

### Mistake 4: Picking the AWS Service Without Reading the Scenario

'Which service provides scalability?' Options: ALB, Auto Scaling, CloudFront, EBS.

ALB = load balancing (not scaling). Auto Scaling = automatic scaling (yes). CloudFront = caching (not scaling compute). EBS = storage (not scaling).

The keyword is 'automatic' or 'elastic'; Auto Scaling is the only one.

### Mistake 5: Overcomplicating IAM

The exam tests IAM at a high level: users vs roles, root MFA, least privilege.

Don't overthink policy syntax; know concepts: roles are better than keys for services, MFA on root, least privilege principle.
