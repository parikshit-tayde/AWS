# Technology & Billing Review — Domains 3 & 4

## Overview

Domains 3 and 4 comprise 49% of the AWS Certified Cloud Practitioner exam. Domain 3 (Technology and Services) accounts for 33% (~50 questions) and tests knowledge of AWS services and when to use each. Domain 4 (Billing, Pricing, and Support) accounts for 16% (~24 questions) and focuses on cost models, support plans, and billing concepts.

This review guide connects the 19 technical topics from earlier in the series to their exam application and bundles them with pricing and support concepts.

## Domain 3: Technology and Services (33% of exam, ~50 questions)

### Compute Services

**Amazon EC2 (Elastic Compute Cloud)**
- IaaS offering: you control OS, runtime, application
- When to use: full control needed, OS-level customization required, long-running workloads
- Instance types: General (t3, m5), Compute-optimized (c5), Memory-optimized (r5, x1), GPU (p3), Storage-optimized (i3)
- Lifecycle: stopped (retain storage), terminated (delete everything), rebooted (keeps all data)
- Related Topics: 11, 14, 17, 18 (Multi-AZ, Auto Scaling, load balancing)

**AWS Lambda**
- Serverless compute: AWS manages infrastructure, customer uploads code
- Billing: pay per invocation + compute duration (milliseconds); free tier 1M invocations/month
- When to use: event-driven workloads (S3 uploads, API calls, scheduled tasks), no servers to manage
- Cold start: ~10-30ms on first invocation after period of inactivity
- Related Topics: 12 (event-driven patterns)

**Elastic Beanstalk**
- PaaS offering: upload code, Elastic Beanstalk handles OS, runtime, scaling
- When to use: web applications where you want to manage code but not infrastructure
- Abstracts EC2, RDS, load balancing; easier than raw EC2, less control than Lambda
- Related Topics: 11, 14 (deployment patterns)

**When to choose:**
- EC2: full control, complex workloads, specific OS requirements
- Lambda: event-driven, short-running, cost-sensitive, rapid scaling
- Elastic Beanstalk: web apps, known load patterns, team wants to focus on code

### Storage Services

**Amazon S3 (Simple Storage Service)**
- Object storage: store any file type (images, logs, backups, code)
- Pricing: per GB stored, per request, data transfer out
- Durability: 11 nines (99.999999999%); data is redundant across multiple AZs by default
- Use case: static websites, data lakes, backups, log archival, media serving via CloudFront
- Storage classes: Standard (frequent access), Intelligent-Tiering (variable access), Glacier (archive), Glacier Deep Archive (very rare access)
- Related Topics: 13 (CloudFront), 14, 16 (security), 19 (cost optimization)

**Amazon EBS (Elastic Block Store)**
- Block storage: attached to EC2 instances; persists independently of instance
- Pricing: per GB-month, per IOPS
- Use case: primary storage for EC2, databases, highly random I/O
- EBS snapshots: point-in-time backups; can be used to create new volumes
- Snapshot lifecycle policies can auto-delete old snapshots (cost control)
- Related Topics: 11, 18 (high availability)

**Amazon EFS (Elastic File System)**
- File storage: shared filesystem mounted by multiple EC2 instances
- Pricing: per GB stored, per request
- Use case: shared data across instances, NFS-compatible
- Difference from S3: low-latency shared access; not for archival

**When to choose:**
- S3: any file, any size, global access, long retention, cost-optimized tiers
- EBS: block storage for EC2, databases, random I/O, snapshot backup
- EFS: shared filesystem for multiple EC2 instances

### Database Services

**Amazon RDS (Relational Database Service)**
- Managed relational database: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server
- PaaS: AWS handles backups, patching, failover
- Pricing: instance type + storage + backups + data transfer
- Multi-AZ: automatic failover to standby replica in different AZ (~60s recovery)
- Read replicas: scale read traffic; eventual consistency
- Use case: structured data, ACID compliance needed, joins and transactions
- Related Topics: 12, 14, 18 (reliability)

**Amazon DynamoDB**
- NoSQL database: key-value and document storage
- Serverless: no capacity planning; scales automatically
- Pricing: read/write capacity units (provisioned) or per-request (on-demand)
- Use case: real-time applications, IoT, gaming leaderboards, sessions, high-scale, variable load
- Eventual consistency by default; strong consistency optional
- Related Topics: 12 (when to use vs RDS)

**Amazon ElastiCache**
- In-memory cache: Redis or Memcached
- Use case: cache frequently accessed data to reduce database load and latency
- Pricing: node type + storage
- Related to RDS: sits in front to reduce database queries

**When to choose:**
- RDS: structured data, ACID transactions, complex queries, joins
- DynamoDB: document/key-value, real-time, auto-scaling, simple queries
- ElastiCache: caching layer above RDS or DynamoDB

### Networking and CDN

**Amazon VPC (Virtual Private Cloud)**
- Isolated network within AWS; contains subnets, security groups, route tables
- Public subnet: instances get public IPs; can reach internet via Internet Gateway
- Private subnet: no public IPs; outbound via NAT Gateway; inbound only from within VPC
- Security Group: stateful firewall on instance level (allow/deny rules)
- Network ACL: stateless firewall on subnet level (allow/deny rules)
- Related Topics: 10 (security groups), 13 (routing)

**Amazon Route 53**
- DNS service: route traffic based on latency, failover, weighted, geolocation
- Health checks: continuous monitoring; automatic failover if primary is unhealthy
- Use case: global traffic distribution, disaster recovery, A/B testing
- Related Topics: 13, 18 (failover routing)

**Amazon CloudFront**
- CDN: edge locations globally; cache content close to users
- Pricing: data transfer out, per-request, HTTPS (small surcharge)
- Use case: static website, video streaming, API acceleration, DDoS protection
- Origin Shield: additional caching layer between edge and origin (costs more, improves cache hit ratio)
- Related Topics: 13, 14

### Monitoring and Logging

**Amazon CloudWatch**
- Metrics: CPU, memory, disk, network from EC2, RDS, Lambda, etc.
- Logs: application logs, system logs, centralized log aggregation
- Alarms: threshold-based alerts; trigger actions (SNS, Auto Scaling, Lambda)
- Dashboards: visualize metrics and logs
- Related Topics: 15 (observability)

**AWS CloudTrail**
- Audit trail: logs all API calls made to AWS
- Use case: compliance, troubleshooting, security investigation
- 90-day free history; long-term storage to S3
- Related Topics: 16 (security, compliance)

**AWS Trusted Advisor**
- Automated checks: security, cost, reliability, performance, quotas
- Free tier: 5 checks; Business/Enterprise: 115 checks
- Use case: continuous optimization, catching common mistakes
- Related Topics: 19 (cost optimization, best practices)

### High Availability and Resilience (from Topics 10, 17, 18)

**Elastic Load Balancer (ALB)**
- Distributes traffic across targets in multiple AZs
- Health checks: removes unhealthy targets automatically
- Session persistence: sticky sessions via cookies
- Related Topics: 10, 18

**Auto Scaling Group**
- Maintains desired number of instances
- Replaces unhealthy instances automatically
- Scales out on high CPU/traffic; scales in when load drops
- Target Tracking: keep metric (CPU, request count) at target value
- Related Topics: 10, 18

**Multi-AZ Deployments**
- RDS Multi-AZ: automatic failover to standby replica
- ALB + ASG across AZs: geographic fault tolerance
- Related Topics: 18, 19

## Domain 4: Billing, Pricing, and Support (16% of exam, ~24 questions)

### AWS Pricing Models

**On-Demand**
- Pay per second/hour of usage; no commitment
- Highest per-unit cost; most flexible
- Use case: variable load, unpredictable demand, short-lived workloads

**Reserved Instances (RI)**
- 1-year or 3-year commitment; 30-60% discount vs on-demand
- Upfront payment or partial upfront
- Use case: steady-state workloads with predictable usage
- Cannot be cancelled; unused capacity is still charged

**Spot Instances**
- 70-90% discount vs on-demand; AWS can reclaim with 2-minute notice
- Use case: batch jobs, non-critical workloads, fault-tolerant applications
- Cannot be used for RDS or databases

**Savings Plans**
- 1-year or 3-year commitment to spending amount (e.g., $1000/year)
- Applies across instance families, sizes, regions (flexible)
- 15-50% discount vs on-demand
- More flexible than RIs; can be shared across accounts

**Free Tier**
- 12 months: 750 hours EC2 t2.micro/t3.micro, 5GB S3 standard, RDS db.t2.micro single-AZ
- Always free: 1M Lambda invocations, 1GB DynamoDB storage
- Use case: learning, development, testing

### Cost Optimization

**Common savings opportunities:**
- Identify idle EC2 instances (Trusted Advisor)
- Right-size RDS instances (downsize over-provisioned databases)
- Use spot instances for non-critical workloads
- Enable S3 Intelligent-Tiering (automatically move to cheaper tiers)
- Delete unattached Elastic IPs (0.005/hour each)
- Use Reserved Instances or Savings Plans for predictable workloads
- Archive old logs to S3 Glacier

**Trusted Advisor recommendations:**
- Underutilized RDS instances (often 5-20% savings)
- Reserved instance opportunities (commit to patterns you see)
- Unattached Elastic IPs (3.65/month each)

### AWS Support Plans

| Plan | Cost | Response time (prod) | Architectural guidance | Trusted Advisor | TAM | Use case |
|---|---|---|---|---|---|---|
| Basic | Free | Not included | Community forums | 5 checks | No | Learning, non-prod |
| Developer | 29/mo | 12-24 hours | Best effort | 5 checks | No | Dev/test environments |
| Business | 100+/mo | 1 hour | Contextual | All 115 | No | Production workloads |
| Enterprise | 15,000+/mo | 15 minutes | Proactive | All 115 | Yes | Mission-critical, 24/7 |

### AWS Organizations and Consolidated Billing

**AWS Organizations:**
- Manage multiple AWS accounts from a central master account
- Organization Units (OUs): logical groupings for policy application
- Service Control Policies (SCPs): restrict maximum permissions (e.g., prevent s3:DeleteBucket)
- Consolidated billing: aggregate costs across accounts, volume discounts apply
- Use case: large enterprises, multi-team deployments, cost control across accounts

**Consolidated Billing benefits:**
- Volume discounts aggregate across accounts
- Reserved instances can be shared (any account in the organization)
- Single bill for all accounts
- Cost allocation tags for chargeback

## Exam Patterns for Domain 3 & 4

### Pattern 1: Service Selection

**Question:** 'A company needs to store 500GB of log files that are accessed daily for 30 days, then rarely accessed. Which service is most cost-effective?'

**Answer:** S3 Intelligent-Tiering (automatically moves to cheaper storage after 30 days) or S3 Lifecycle policy (move to Glacier after 30 days).

### Pattern 2: Pricing Optimization

**Question:** 'A company runs a database with consistent, predictable load year-round. Which pricing model saves the most?'

**Answer:** Reserved Instance or Savings Plan (30-60% discount vs on-demand).

### Pattern 3: High Availability

**Question:** 'A company wants to ensure the database survives an AZ failure. Which feature should they enable?'

**Answer:** Multi-AZ (for RDS) or read replica with automatic failover.

### Pattern 4: Support Plan Selection

**Question:** 'A SaaS startup runs production workloads on AWS. The cost of a 1-hour outage is ~10,000 dollars. Which support plan should they choose?'

**Answer:** Business support (1-hour response time, 100-200/mo cost << 10,000 outage risk).

## Full-Length Practice Exam Strategy

### Before the exam

1. Take a 130-question full-length practice exam (Udemy, Tutorials Dojo)
2. Time yourself: 90 minutes (41 seconds per question)
3. Do NOT look up answers during the exam; flag and move on
4. Goal: 75%+ (98 out of 130 correct)

### Review your practice exam

1. Score breakdown by domain: where did you lose points?
2. Domain 3 weakness? Review the service comparison tables above
3. Domain 4 weakness? Review pricing models and support plans
4. Redo the questions you got wrong; understand why you missed them

### Exam day preparation

1. Get 7+ hours of sleep the night before
2. Eat a light meal 1-2 hours before exam
3. Arrive 15 minutes early; no notes allowed
4. Read each question carefully; underline key words
5. Flag difficult questions; answer all easy ones first
6. Return to flagged questions in remaining time
7. Do not second-guess; trust your training

### When to take the real exam

- After passing 2-3 full-length practice exams at 80%+
- After scoring 85%+ on all 5-question domain quizzes
- After reviewing all weak topics
- Passing Cloud Practitioner is achievable with this series; it is the foundational cert
